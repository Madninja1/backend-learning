# Lifecycle PHP запроса

### Цепочка обработки запроса:

* Browser ↓
* Nginx ↓
* PHP-FPM ↓
* Zend Engine ↓
* Response

---

## Browser
1. Сначала браузер делает DNS lookup — запрос к DNS-серверу, 
чтобы узнать IP адрес домена.

> example.com → 93.184.216.34

Если сайт открывался недавно, браузер или операционная система 
могут использовать DNS-кэш, и тогда повторный DNS запрос не выполняется.

После получения IP браузер устанавливает TCP соединение с сервером.

Далее:

> HTTP  → порт 80
>
> HTTPS → порт 443

HTTPS — это HTTP поверх TLS, то есть соединение шифруется с помощью TLS.

После установки соединения браузер отправляет HTTP request на сервер.

---

## Nginx
Nginx — это web server и reverse proxy, который принимает входящие HTTP запросы.

Основные задачи nginx:

**1. Обработка соединений**

Nginx эффективно работает с большим количеством сетевых соединений:

- держит тысячи подключений
- управляет keep-alive соединениями
- обрабатывает HTTP запросы

Это важно, потому что PHP сам плохо работает с большим количеством соединений.

**2. Отдача статических файлов**

Nginx может сам отдавать статические файлы без запуска PHP:

> css
>
>js
>
>images
>
>fonts
>
>videos

Это значительно ускоряет работу сайта.

**3. Передача запроса в PHP**

Если запрос требует выполнения PHP кода, nginx передаёт его в 
php-fpm через протокол FastCGI.

Обычно входная точка приложения — это файл:

> public/index.php

---

## php-fpm
PHP-FPM (FastCGI Process Manager) — это менеджер процессов для PHP.

Он запускает **пул воркеров (worker processes)**, 
которые обрабатывают PHP запросы.

Когда nginx передаёт запрос через FastCGI:

1. PHP-FPM выбирает свободный worker
2. worker получает путь к PHP файлу
3. запускается выполнение PHP кода

Использование пула процессов позволяет:

- обрабатывать несколько запросов одновременно
- изолировать память между запросами
- повысить стабильность системы

---

## Zend Engine

Zend Engine — это **ядро интерпретатора PHP**, которое выполняет PHP код.

Процесс выполнения выглядит примерно так:

```
PHP code
↓
Lexer (разбиение на токены)
↓
Parser
↓
AST (Abstract Syntax Tree)
↓
Opcode
↓
Execution
```

PHP код **не компилируется напрямую в машинный код**.

Он компилируется в **opcode (байткод)**, 
который затем исполняется виртуальной машиной Zend Engine.

---

## OPcache

Для ускорения работы используется OPcache.

Он сохраняет скомпилированный opcode в памяти, 
чтобы не компилировать PHP файл заново при каждом запросе.

---

## Response
После выполнения PHP кода формируется **HTTP response**.

Этот ответ проходит обратный путь:
```
Zend Engine
↓
PHP-FPM
↓
Nginx
↓
Browser
```
Nginx отправляет HTTP response обратно клиенту 
и браузер отображает результат пользователю.

---

Каждый HTTP запрос к PHP приложению — это отдельный процесс выполнения.

PHP не хранит состояние между запросами, поэтому для хранения данных используются:
```
database
redis
cache
sessions
```
